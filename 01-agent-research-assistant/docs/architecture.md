## 4. 关键技术实现

### 4.1 ReAct 推理引擎

ReAct 引擎是整个系统的核心，实现了 **Thought → Action → Observation** 的循环推理模式。

```typescript
// 简化伪代码
class ReActEngine {
  async run(task: string): Promise<string> {
    for (let i = 0; i < maxIterations; i++) {
      // 1. 构建 prompt（包含工具描述 + 历史步骤）
      const prompt = this.buildPrompt(task, previousSteps);
      
      // 2. 调用 LLM 生成下一步
      const response = await this.llm.generate(prompt);
      
      // 3. 解析响应（支持多种格式）
      const step = this.parseResponse(response);
      
      // 4. 如果是最终答案，返回
      if (step.finalAnswer) return step.finalAnswer;
      
      // 5. 如果是工具调用，执行工具
      const observation = await this.tools.execute(step.action);
      step.observation = observation;
    }
  }
}
```

### 4.2 响应解析策略

为兼容不同 LLM 的输出格式，实现了多种解析器：

| 格式 | 示例 | 解析方式 |
|------|------|----------|
| 标准格式 | `Action: tool[{"query":"..."}]` | 正则匹配 |
| 括号格式 | `Action: tool({"query":"..."})` | 正则匹配 |
| 冒号格式 | `Action: tool: {"query":"..."}` | 正则匹配 |
| 中文格式 | `行动：tool[{"query":"..."}]` | 正则匹配 |
| 兜底方案 | 包含工具名的任意文本 | 工具名匹配 |

### 4.3 SSE 流式传输

```typescript
// 后端：设置 SSE 响应头
res.setHeader('Content-Type', 'text/event-stream');
res.setHeader('Cache-Control', 'no-cache');
res.setHeader('Connection', 'keep-alive');

// 发送数据
res.write(`data: ${JSON.stringify(step)}\n\n`);

// 前端：读取 SSE 流
const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  
  const chunk = decoder.decode(value);
  const lines = chunk.split('\n');
  
  for (const line of lines) {
    if (line.startsWith('data: ')) {
      const data = JSON.parse(line.slice(6));
      // 处理数据...
    }
  }
}
```

## 6. 关键设计决策

### 6.1 为什么选择 ReAct 而非纯 Function Calling?

| 方案 | 优点 | 缺点 |
|------|------|------|
| **ReAct（本项目）** | 兼容所有 LLM、推理过程透明、可解释性强 | 文本解析不稳定、Prompt 工程复杂 |
| **Function Calling** | 结构化输出、可靠、无需解析 | 仅支持部分模型、黑盒 |

**建议**: 生产环境优先使用 Function Calling，ReAct 作为降级方案。

### 6.2 为什么选择 SSE 而非 WebSocket?

| 方案 | 优点 | 缺点 |
|------|------|------|
| **SSE（本项目）** | 浏览器原生支持、HTTP 兼容、自动重连 | 单向通信、连接数限制 |
| **WebSocket** | 全双工、低延迟、无连接数限制 | 协议升级复杂、需心跳保活 |

**建议**: Agent 单向推送场景 SSE 足够，需要双向交互时用 WebSocket。

### 6.3 为什么选择 SQLite 而非 Redis / PostgreSQL?

| 方案 | 优点 | 缺点 |
|------|------|------|
| **SQLite（本项目）** | 零配置、嵌入式、无需运维 | 不支持高并发、不适合分布式 |
| **Redis** | 高性能、支持 pub/sub、TTL | 内存成本高、数据持久化需配置 |
| **PostgreSQL + pgvector** | 功能强大、支持向量检索 | 需要外部服务、运维成本高 |

**建议**: 单机部署用 SQLite，生产环境用 PostgreSQL + pgvector。
